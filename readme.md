• Обзор

  - Проект — это небольшая PWA-панель TeamsHGT: HTML обёртка подключает один модуль app-shell и запускает ES-модульный бандл
    (index.html:13, index.html:14), а всё поведение реализовано на чистых Web Components плюс сервис-воркер для офлайн-режима.
  - manifest.json и service-worker.js описывают нужные иконки, тему и стратегию кэширования для установки и работы офлайн; сервис-воркер
    кладёт все статики в teamshgt-v1 и обновляет их при активации (service-worker.js:1-52).

  Слои и компоненты

  - Входной модуль scripts/main.js импортирует все компоненты и берёт на себя только PWA-хуки и глобальные события (scripts/main.js:1-48),
    изолируя инфраструктуру от UI.
  - AppShell берёт на себя маршрутизацию по hash, генерацию navigation UI из конфигурационной карты и пробрасывает кастомные события на
    установку/обновление (scripts/components/app-shell.js:1-99), все остальные области — независимые элементы.
  - Каталог сотрудников и виджеты работают автономно: directory-view фильтрует общие данные из scripts/data/employees.js и отображает
    карточки (scripts/components/directory-view.js:1-81); attendance-tools хранит локальный список активностей и рендерит формы и ленту
    (scripts/components/attendance-tools.js:1-111); news-board и profile-insights читают свои данные из изолированных модулей (scripts/
    components/news-board.js:1-35, scripts/components/profile-insights.js:1-55).
  - Вся конфигурация и мок-данные лежат отдельно в scripts/data, что упростит замену на реальные API (scripts/data/employees.js:1-69,
    scripts/data/news.js:1-23).
  - Стилевой слой централизован, использует CSS custom properties для палитры и общего набора layout-утилит (styles/main.css:1-80),
    поэтому новые блоки автоматически наследуют тему.

  DRY / SOLID / KISS
  - Связь между слоями реализована событием CustomEvent вместо прямых вызовов (например, AppShell диспатчит teams:refresh, а слушатель
    висит в main.js), что снижает связность и позволяет заменить обработчики в будущем (scripts/components/app-shell.js:69-75, scripts/
    main.js:34-48).
  - PWA-инфраструктура инкапсулирована: сервис-воркер обслуживает только GET-запросы и обновляется атомарно, так что можно добавить новые
    ресурсы без изменения клиентской логики (service-worker.js:34-51).
  - Масштабирование: чтобы добавить новый раздел, достаточно создать новый компонент, положить его рядом и добавить запись в viewMap — все
    кнопки и маршрутизация построятся автоматически (scripts/components/app-shell.js:34-67); данные можно переключить на fetch/REST, меняя
    только модуль scripts/data/*.
  - Гибкость UI обеспечивается тем, что стилевые токены и утилиты (styles/main.css:65-80) уже покрывают адаптивность и тёмную тему, а
    компоненты работают через шаблоны строк, поэтому новые поля/карточки легко добавить без ломки остальных областей.

  Следующие шаги

  1. Если планируется реальное подключение к бэкенду — заменить файлы scripts/data на сервисы с fetch/GraphQL и добавить обработку
     загрузки/ошибок.
  2. Подумать о юнит-тестах для компонентов (например, через Web Test Runner) перед тем как наращивать функциональность.
